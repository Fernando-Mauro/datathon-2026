## Deviations from PLAN

### 1. Sandbox blocked `npx ampx` — orchestrator ran the deploy

Same documented Phase 1 fallback path: executor's bash sandbox blocks `npx ampx sandbox` even with `dangerouslyDisableSandbox: true`. Orchestrator ran the deploy directly. Functionally equivalent.

### 2. AWS introspection commands denied (acceptance criteria substitution)

Plan 02-04's acceptance criteria included `aws cognito-idp list-user-pools` and `aws cloudformation describe-stack-resource` as post-deploy proofs. Both fail with `AccessDenied` because `AmplifyBackendDeployFullAccess` is narrowly scoped to *deploy* operations — it does NOT include read permissions like `cognito-idp:ListUserPools` or `cloudformation:DescribeStackResource`.

**Substitute evidence (equivalent or stronger):**
- The deploy itself emitted live CloudFormation events showing `CREATE_COMPLETE` for the UserPool — direct proof of resource creation
- `amplify_outputs.json` was regenerated by AWS itself (not a fake) and contains the real `user_pool_id` (`us-east-1_6l4dSfRCz`) which AWS only emits after the resource exists
- Phase 5 (production deploy) will likely need to widen IAM permissions for verification, OR we accept that deploy-time evidence + outputs file are sufficient. Decision deferred to Phase 5 planning.

This is documented as an accepted deviation, not a blocker. AUTH-01 verification path is sound.

### 3. Faster than estimated

RESEARCH estimated 2-3 min for incremental update; actual was 45 seconds. CDK Bootstrap from Phase 1 + small auth resource = trivial deploy. No action needed; faster is better.
