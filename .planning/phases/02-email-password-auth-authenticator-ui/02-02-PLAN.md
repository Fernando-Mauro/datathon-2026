<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| npm registry → developer machine | `bun add` downloads tarballs from npm; integrity is verified via the `bun.lock` SHA-512 hashes (Bun's default behavior). |
| Installed node_modules → application bundle | Once installed, `aws-amplify` + `@aws-amplify/ui-react` ship in the production browser bundle (they are runtime deps). Any vulnerability in this tree is a real frontend exposure. |
| Audit ignore-list → dependency CVE acceptance | The `--ignore=` flags accept specific advisories as known-tolerated risks. Each ignore is a explicit risk acceptance. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-2-02-01 | T (Tampering) | npm package integrity | mitigate | bun.lock pins SHA-512 hashes; reproducible installs. |
| T-2-02-02 | I (Information Disclosure) | New CVEs in `aws-amplify` v6 transitives | mitigate (or accept-with-justification) | `bun run audit` is run raw first; any new advisory either gets fixed (via `bun update` / `overrides`) or accepted with a one-line justification recorded in SUMMARY. No silent ignores. |
| T-2-02-03 | E (Elevation of Privilege) | Malicious package post-install scripts | mitigate | `package.json#ignoreScripts: ["sharp", "unrs-resolver"]` carries forward from Phase 1; new Amplify v6 packages have NO native binaries (per L-13), so no `trustedDependencies` extension needed. |
| T-2-02-04 | I | Bundle-size growth (~30KB+ gzip Amplify CSS + JS) | accept | Hackathon-acceptable; documented in RESEARCH §8 ("acceptable for a hackathon scaffold"). |
</threat_model>
