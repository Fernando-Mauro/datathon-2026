### `amplify/auth/resource.ts` (CREATE — backend-resource, CDK construct factory)

**Analog:** `amplify/backend.ts` (Phase 1) — same `import { defineX } from "@aws-amplify/backend"` + factory-call shape. Also serves as the only existing reference for the file-header JSDoc convention sanctioned in Phase 1 PATTERNS for Amplify backend files.

**Imports pattern from `amplify/backend.ts:1` (mirror — single named import, double quotes):**
```typescript

```

**File-header JSDoc pattern from `amplify/backend.ts:3-7` (mirror — multi-line JSDoc explaining intent + `@see` link):**
```typescript
/**
 * Phase 1: Bare backend skeleton — no resources yet.
 * Phase 2 will add auth (Cognito + email/password).
 * @see https://docs.amplify.aws/nextjs/build-a-backend/
 */
```
Phase 1 PATTERNS sanctioned this as "the **only** sanctioned multiline-comment block" — Phase 2's `auth/resource.ts` qualifies as the same kind of file (Amplify backend module documenting intent for future phases) and inherits the same allowance.

**Core pattern** (RESEARCH §Code Examples §1, verbatim — already in double-quote scaffold style):
```typescript
import { defineAuth } from "@aws-amplify/backend";

/**
 * Phase 2 — first backend resource on top of the Phase 1 skeleton.
 * Cognito User Pool with email + password sign-in, mandatory email verification,
 * and email-based password reset. Default password policy (min 8, upper+lower+digit+symbol).
 *
 * Attribute name note: the CDK property `fullname` maps to the Cognito/OIDC standard
 * claim `"name"`. The <Authenticator> picks this up via Zero Configuration from
 * amplify_outputs.json and auto-renders a "Name" field in the sign-up form.
 *
 * @see https://docs.amplify.aws/react/build-a-backend/auth/
 */
export const auth = defineAuth({
  loginWith: { email: true },
  userAttributes: {
    email: { required: true, mutable: false },
    fullname: { required: true, mutable: true },
  },
  accountRecovery: "EMAIL_ONLY",
});
```

**Gotchas applying to this file:**
- L-1: CDK property is `fullname` (NOT `name`). Writing `name: { ... }` causes a TypeScript "Object literal may only specify known properties" error. **The wire-format Cognito attribute name `"name"` only appears in `amplify_outputs.json` — never in this source file.**
- RESEARCH §1: `email: { required: true, ... }` is technically redundant (Amplify auto-requires email when `loginWith.email = true`), but `mutable: false` is meaningful. Keep both for clarity.
- RESEARCH §1: Do NOT add `passwordPolicy` — the default already matches CONTEXT D-20.
- RESEARCH §1: Do NOT add `userVerification` — default `emailStyle = CODE` matches D-19.
- Quote style: double quotes (Phase 1 PATTERNS rule, satisfied by RESEARCH excerpt as-is).
- ESM-loaded by `tsx` via the existing `amplify/package.json` (`{"type": "module"}`) — no separate package.json needed for the `auth/` subdirectory.

**Core pattern** (RESEARCH §Code Examples §5, verbatim — already double-quote style):
```tsx
"use client";

import { useEffect } from "react";
import { useRouter } from "next/navigation";
import { Authenticator, useAuthenticator } from "@aws-amplify/ui-react";

export default function LoginPage() {
  const router = useRouter();
  const { authStatus } = useAuthenticator((context) => [context.authStatus]);

  useEffect(() => {
    if (authStatus === "authenticated") {
      router.push("/app");
    }
  }, [authStatus, router]);

  return (
    <main className="flex min-h-screen items-center justify-center p-8">
      {authStatus === "configuring" && (
        <p className="text-sm opacity-70">Loading…</p>
      )}
      {authStatus !== "authenticated" && <Authenticator />}
    </main>
  );
}
```

**Gotchas applying:**
- L-4: `useAuthenticator` requires `Authenticator.Provider` ancestor — guaranteed by `AmplifyProvider` mounted in root `app/layout.tsx`.
- L-6: Do NOT pass `signUpAttributes` prop. Zero Configuration handles the Name field via `amplify_outputs.json` `standard_required_attributes: ["email", "name"]`.
- L-7: `useRouter` MUST be from `next/navigation` (NOT `next/router`, which is Pages Router only). Easy training-data trap.
- L-15: `[authStatus, router]` dependency array is exhaustive under `react-hooks/exhaustive-deps`. Do NOT add eslint-disable comments.
- RESEARCH §6: Selector form `(context) => [context.authStatus]` is required to prevent re-renders on every form keystroke. Without it, the page re-renders excessively but still functions.
- RESEARCH §6: `authStatus === "configuring"` is a transient state during initial cookie-rehydration mount; the loading message is brief.
- Quote style: double quotes throughout.

**Core pattern** (RESEARCH §Code Examples §7, verbatim — double-quote style preserved):
```tsx
"use client";

import { useRouter } from "next/navigation";
import { signOut } from "aws-amplify/auth";

export function SignOutButton() {
  const router = useRouter();

  async function handleClick() {
    await signOut();
    router.push("/");
  }

  return (
    <button
      type="button"
      onClick={handleClick}
      className="rounded-md border border-foreground/20 px-4 py-2 text-sm hover:bg-foreground/5"
    >
      Sign out
    </button>
  );
}
```

- L-7: `useRouter` from `next/navigation`.
- L-14: `await signOut()` is mandatory **before** `router.push("/")` — without await, the cookie may persist past navigation.
- RESEARCH §7.a: import path is `"aws-amplify/auth"` (v6 subpath) — NOT `import { Auth } from "aws-amplify"` (v5 namespace, gone in v6).
- RESEARCH §7.b: using `signOut` from `aws-amplify/auth` directly (vs. the `signOut` from `useAuthenticator()`) avoids the need for a `Authenticator.Provider` ancestor in `app/app/` — this is one of the reasons the root-layout hoisting strategy works cleanly.
- Named export (NOT default) because it's imported by name in `app/app/page.tsx`.
- `type="button"` is best-practice (avoids accidental form submission if ever placed inside a `<form>`).
