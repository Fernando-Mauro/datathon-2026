<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → Bundled CSS | User-controlled style/className values eventually flow through PostCSS at build time; a CVE in PostCSS's `</style>` stringify path could let crafted CSS escape the `<style>` boundary at runtime. |
| Developer machine → Registry | `bun update postcss` and `bun add -D` pull artifacts from the public npm registry; lockfile pinning is the integrity boundary. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-1-01 | Tampering / Information Disclosure | `postcss` (transitive of `next` and `@tailwindcss/postcss`) — GHSA-qx2v-qp2m-jg93 (PostCSS XSS via unescaped `</style>` in stringify output, CVE-2026-41305) | mitigate | Bump lockfile entry to `postcss >= 8.5.10` via `bun update postcss`; verify with `bun audit --audit-level=moderate` exiting 0. Defensive fallback: `bun add -D postcss@latest` if the transitive bump does not lift the version. |
| T-1-02 | Tampering | Lockfile-out-of-sync with `node_modules/` after partial command (G-14) | mitigate | Run `bun install` defensively after every `bun update`/`bun add` so the materialized tree matches the lockfile. |
| T-1-03 | Repudiation | Pre-existing lint warning regresses through `--max-warnings=0` gate without notice | mitigate | Run `bun run lint` immediately after the script change as a smoke pass; any warning surfaced now (instead of in a later plan) prevents silent CI bypass later. |

Note: Phase 1 introduces no auth, no user input, and no networked endpoints inside the app, so the realistic threat surface is small. The PostCSS CVE is the one concrete pre-existing high-impact issue and INFRA-03's mitigation for it is the work of this plan.
</threat_model>
