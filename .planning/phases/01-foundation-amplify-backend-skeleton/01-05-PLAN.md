<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Developer machine → AWS account | `npx ampx sandbox` uses long-lived (or SSO-issued) IAM credentials from `~/.aws/credentials` to perform `cloudformation:*`, `iam:*`, and CDK-bootstrap S3/ECR operations. The IAM principal is the trust anchor for everything deployed. |
| Synthesized CFN template → CloudFormation service | CDK-synthesized templates are uploaded to the CDK bootstrap S3 bucket and applied by CloudFormation; an unintended construct in `backend.ts` would deploy unintended infra. |
| Generated `amplify_outputs.json` → repo | The file contains stack-specific resource IDs and (in later phases) public OAuth client IDs. Must NOT be committed. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-1-19 | Elevation of Privilege | Developer attaches `AdministratorAccess` instead of `AmplifyBackendDeployFullAccess`, dramatically expanding the blast radius if the credential leaks | mitigate (documentation) | Plan 04 README §Setup explicitly names `AmplifyBackendDeployFullAccess` as the recommended policy. This plan re-references that doc in the manual gate. AWS-side enforcement is out of scope (no central org policy under our control). |
| T-1-20 | Information Disclosure | `amplify_outputs.json` accidentally committed in a `git add .` | mitigate | Plan 02's `.gitignore` adds `amplify_outputs*` (no path prefix); this plan's acceptance criterion runs `git check-ignore -q amplify_outputs.json` to PROVE the pattern works. |
| T-1-21 | Information Disclosure | Logging of `~/.aws/credentials` contents anywhere in execution output | mitigate | Action explicitly forbids printing the credentials file; only `aws sts get-caller-identity` (which returns Account ID + ARN, not secrets) is invoked. The resume-signal asks the developer to paste the JSON output, which by design contains no secret material. |
| T-1-22 | Tampering / Elevation of Privilege | `amplify/backend.ts` modified between plan 03 and this plan to add unintended resources, leading to surprise infra in the deploy | mitigate | Acceptance criteria assert `amplify/backend.ts` is unchanged from plan 03 (still exactly `defineBackend({})`). The verifier diff against the SUMMARY from plan 03 catches drift. |
| T-1-23 | Denial of Service (cost) | Sandbox stack left running indefinitely accrues minor CDK bootstrap charges (S3 storage + ECR repo) | accept | Empty backend has near-zero ongoing cost; CDK bootstrap S3 bucket is sub-MB. Documented cleanup command (`npx ampx sandbox delete`) is in README §Setup step 7 and surfaced in the SUMMARY for the developer to invoke at their discretion. |
| T-1-24 | Tampering | Watch-mode launched accidentally (without `--once`) and left running, hot-deploying every save | mitigate | Action explicitly uses `--once`; hard rule forbids running without it. |

Note: This is the only Phase 1 plan that actually touches AWS. The threats above are the realistic ones for an empty-backend deploy. The CVE/lint/build threats from plans 01-04 are still mitigated by their respective plans; this plan re-runs those gates as smoke checks.
