<gauntlet_command>
The Phase 1 verification gauntlet is a single shell pipeline:

```bash
bun run lint && bun run typecheck && bun run audit && bun run build
```

Order matters:
1. `bun run lint` — fastest layer; catches double-quote violations, exhaustive-deps issues, no-console (the `// eslint-disable-next-line` comment in `app/app/page.tsx` Task 2 is a known exception)
2. `bun run typecheck` — slower; catches type errors after lint passes (e.g., a typo in `safeFromPath` signature)
3. `bun run audit` — checks the `bun.lock` against the CVE database; with the established 24-entry ignore-list (Phase 1 plan 01-03), should report no NEW vulnerabilities. Phase 3 added zero deps so the ignore-list does not grow.
4. `bun run build` — slowest (~30s); the load-bearing layer for Phase 3 because it catches the L-1 / K-3 Suspense regression that dev-mode silently passes. If `app/login/page.tsx` were missing the `<Suspense>` wrapper, this step would fail with: `Error: useSearchParams() should be wrapped in a suspense boundary at page "/login". Read more: https://nextjs.org/docs/messages/missing-suspense-with-csr-bailout`.

The `&&` chain ensures the pipeline stops at the FIRST failing step — that surfaces the most actionable error early.
</gauntlet_command>
