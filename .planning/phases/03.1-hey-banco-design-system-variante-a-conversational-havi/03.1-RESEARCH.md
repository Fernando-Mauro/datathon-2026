# Phase 03.1: Hey-banco design system + Variante A conversational HAVI — Research

**Researched:** 2026-04-25
**Domain:** UI/UX foundation (typography, color system, custom Authenticator theme) + conversational chat shell + nested route transitions on top of Next 16.2.4 + React 19.2 + Tailwind 4.2 + Amplify UI 6.15.3 + framer-motion (motion@12.38)
**Confidence:** HIGH for all installed-package APIs (Amplify theme, next/font, Tailwind 4 `@theme inline`, Next 16 client layouts/Suspense). MEDIUM for `motion@12` route-transition pattern (verified against motion.dev docs + npm registry; not yet installed).

## Summary

Phase 03.1 is a **visual replatform plus shell expansion**. It does not add backend, change auth flows, or extend `<AuthGuard>`. It (1) swaps Geist → IBM Plex via `next/font/google` in the root `app/layout.tsx`, (2) ports the 262-line `colors_and_type.css` token set to a single `@theme inline` block in `app/globals.css` (Tailwind 4 generates utility classes automatically), (3) wraps `<Authenticator>` in `/login` with `<ThemeProvider colorMode="dark" theme={...}>` from `@aws-amplify/ui-react` so the login page no longer flashes purple/orange against the dark `/app`, and (4) replaces the Phase 3 placeholder `app/app/page.tsx` with a conversational chat shell whose action cards link out to four new sibling routes (`/app/grafica/[categoria]`, `/app/movimientos`, `/app/comparativa`, `/app/agente`) — all inheriting Phase 3's `<AuthGuard>` for free because the layout is unchanged. Animations come from `motion@12.38` (`framer-motion` renamed mid-2025; new import `motion/react`); HAVI ring stays CSS conic-gradient + `@keyframes` (no JS). localStorage persistence (`havi-chat-v1`) clears on `Hub.listen('auth', …)` `signedOut` event. Pattern-match HAVI lives in `app/_data/patterns.ts`; mock data in `app/_data/mock.ts`; both fully typed.

The biggest non-obvious risk is **`motion@12` is the new package name** (`framer-motion@12` is published as a back-compat shim — both versions are kept in lockstep, but Motion's own upgrade guide directs new installs to `motion`). The next-biggest risk is the **Amplify CSS variables override path**: `<ThemeProvider>` mounts a `<style data-amplify-theme>` block, but the underlying Amplify default stylesheet still gets imported in `app/layout.tsx`, so the CSS-variable cascade order matters (the theme block must load *after* `@aws-amplify/ui-react/styles.css`). The `<Authenticator>` itself does not accept a `theme` prop in v6 — the canonical pattern is `<ThemeProvider theme={…} colorMode="dark"><Authenticator/></ThemeProvider>`.

**Primary recommendation:** Implement the design system in three discrete layers (tokens → fonts → Authenticator theme), then build the chat shell on top. Use `motion/react` (not `framer-motion/`) for new code. Use Tailwind 4 `@theme inline` so all tokens generate utility classes (`bg-hey-bg`, `text-hey-fg-2`, `border-hey-divider`, etc.) — this avoids inline `style={{ background: "var(--hey-bg)" }}` everywhere. Keep HAVI ring as CSS conic-gradient + `@keyframes`. Plan the phase as ~7 plans (see Open Questions §1).

### Pattern 1: Multi-font CSS variable + Tailwind 4 `@theme inline`

**What:** Inject two `next/font/google` CSS variables on `<html>` and map them inside `@theme inline` so Tailwind generates `font-sans` / `font-serif` utilities that resolve to the Plex variables.
**When to use:** Always for typography in this phase. This is the canonical Next.js 16 + Tailwind 4 + Google Fonts pattern.

```tsx
// Source: node_modules/next/dist/docs/01-app/03-api-reference/02-components/font.md (lines 464-580 — "Using Multiple Fonts" with CSS variables)
// app/layout.tsx
import { IBM_Plex_Sans, IBM_Plex_Serif } from "next/font/google";

const plexSans = IBM_Plex_Sans({
  variable: "--font-plex-sans",
  subsets: ["latin"],
  weight: ["400", "500", "700"],
  display: "swap",
});

const plexSerif = IBM_Plex_Serif({
  variable: "--font-plex-serif",
  subsets: ["latin"],
  weight: ["400", "700"],
  display: "swap",
});

export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="es-MX" className={`${plexSans.variable} ${plexSerif.variable} h-full antialiased`}>
      <body className="min-h-full bg-hey-bg text-hey-fg-1 font-sans">
        <AmplifyProvider>{children}</AmplifyProvider>
      </body>
    </html>
  );
}
```


### Pattern 2: Amplify v6 `<ThemeProvider>` wraps `<Authenticator>` (NOT a `theme` prop on `<Authenticator>` directly)

**What:** Build a `Theme` object using `createTheme` (or just a literal Theme object), pass it to `<ThemeProvider theme={…} colorMode="dark">`, render `<Authenticator>` inside.
**When to use:** Once, in `app/login/page.tsx`. Don't put `<ThemeProvider>` at root — it would scope ALL descendants to the Amplify theme system (we want our chat UI on Tailwind, not Amplify primitives).

```tsx
// Source: node_modules/@aws-amplify/ui-react/dist/types/components/ThemeProvider/ThemeProvider.d.ts
//         node_modules/@aws-amplify/ui-react/dist/types/components/Authenticator/Authenticator.d.ts
//         node_modules/@aws-amplify/ui/dist/types/theme/types.d.ts
// app/_styles/amplify-theme.ts
import type { Theme } from "@aws-amplify/ui-react";
import { defaultDarkModeOverride } from "@aws-amplify/ui-react";

export const heyAmplifyTheme: Theme = {
  name: "hey-banco",
  tokens: {
    colors: {
      background: {
        primary: { value: "#000000" },
        secondary: { value: "#1A1A1A" },
      },
      font: {
        primary: { value: "#FFFFFF" },
        secondary: { value: "rgba(255,255,255,0.60)" },
        interactive: { value: "#3478F6" },
      },
      brand: {
        primary: {
          10: { value: "#E8F0FE" },
          20: { value: "#C7DBFD" },
          40: { value: "#7AAAFA" },
          60: { value: "#3478F6" }, // electric blue from D-44
```

### Pattern 3: motion@12 route transitions in App Router via template.tsx OR per-page wrapper

**What:** Per-page motion wrapper component (`<PageTransition>`) is the lowest-friction pattern for our use case. // app/_components/PageTransition.tsx
```tsx
"use client";
import { motion, useReducedMotion } from "motion/react";
import type { ReactNode } from "react";

type Direction = "right" | "bottom";

export function PageTransition({ children, from = "right" }: { children: ReactNode; from?: Direction }) {
  const reduced = useReducedMotion();
  if (reduced) return <>{children}</>;

  const initial =
    from === "right" ? { x: "100%", opacity: 0.4 } : { y: "100%", opacity: 0.4 };
  return (
    <motion.div
      initial={initial}
      animate={{ x: 0, y: 0, opacity: 1 }}
      transition={{ duration: 0.32, ease: [0.32, 0.72, 0, 1] }}
      className="min-h-screen"
    >
      {children}
    </motion.div>
  );
}
```

### Pattern 6: localStorage persistence + sign-out clear via Hub

```ts
// app/_hooks/useChatPersistence.ts (or co-locate inside ChatHome)
"use client";
import { useEffect, useRef } from "react";
import { Hub } from "aws-amplify/utils";
import type { ChatMessage } from "@/app/_data/types";

const KEY = "havi-chat-v1";
const VERSION = 1;

export interface PersistedChat {
  version: number;
  messages: ChatMessage[];
  updatedAt: string;
}

export function loadChat(): ChatMessage[] | null {
  if (typeof window === "undefined") return null;
  try {
    const raw = localStorage.getItem(KEY);
    if (!raw) return null;
    const parsed = JSON.parse(raw) as PersistedChat;
    if (parsed.version !== VERSION) {
      localStorage.removeItem(KEY);
      return null;
    }
    return parsed.messages;
  } catch {
    return null;
  }
}

export function saveChat(messages: ChatMessage[]) {
  if (typeof window === "undefined") return;
  try {
    const payload: PersistedChat = { version: VERSION, messages, updatedAt: new Date().toISOString() };
    localStorage.setItem(KEY, JSON.stringify(payload));
  } catch {
    /* quota exceeded — silently drop, Phase 03.1 is best-effort */
  }
}

export function useSignOutClear() {
  useEffect(() => {
    const stop = Hub.listen("auth", ({ payload }) => {
      if (payload.event === "signedOut") {
        localStorage.removeItem(KEY);
      }
    });
    return stop; // Hub.listen returns the cancellation fn
  }, []);
}
```

### Anti-Patterns to Avoid

- **`<ThemeProvider>` at root layout:** would scope all descendants to Amplify theme tokens; our `/app` UI is Tailwind utility-driven. Mount `<ThemeProvider>` only inside `/login`.
- **`<Authenticator theme={...}>` direct prop:** the `theme` prop does NOT exist on `<Authenticator>` in v6 (verified `node_modules/@aws-amplify/ui-react/dist/types/components/Authenticator/Authenticator.d.ts:8-18`). Wrap with `<ThemeProvider>` (L-1).
- **`Amplify.configure(outputs, { ssr: true })`:** Phase 2 walked back — without `@aws-amplify/adapter-nextjs`, `ssr: true` hangs `useAuthenticator` on `"configuring"`. Phase 03.1 keeps `Amplify.configure(outputs)` (no options).
- **Importing from `framer-motion`:** Phase 03.1 installs `motion@12.38`. Imports MUST be `from "motion/react"` (L-2).
- **HAVI ring as `<img src="/havi-ring.svg">`:** SVG fetch + decode delay; CSS conic-gradient is instant.
- **Mounting `<AnimatePresence>` inside `app/app/layout.tsx`:** layout is persistent, AnimatePresence inside it sees one child forever. Use `template.tsx` if exit animations are needed.
- **Wrapping `<AuthGuard>` in `<motion.div>`:** breaks the redirect timing — guard's `useEffect` may fire before motion's animate-in completes, but more importantly any render-phase animation on a tree that may unmount instantly causes flicker. Animate inside the guarded children, not above.
- **localStorage write on every keystroke:** debounce 250ms (D-47).
- **Inline `style={{ background: "#000" }}` everywhere:** prefer Tailwind utilities generated from `@theme` (`bg-hey-bg`).
- **Re-fetching `fetchUserAttributes()` on every chat-home mount:** cache `firstName` in localStorage (D-50), rehydrate, refresh in background only if stale.

## Common Pitfalls

### Pitfall 1: `<ThemeProvider>` cascade order with default Amplify CSS

**What goes wrong:** The default `@aws-amplify/ui-react/styles.css` defines `:root { --amplify-colors-background-primary: var(--amplify-colors-white); ... }`. `<ThemeProvider>` injects a `<style data-amplify-theme>` block at runtime that re-defines the same variables under `[data-amplify-theme]` selectors. **Specificity:** the theme block has equal or higher specificity if attribute selectors are used. **But import order:** if `app/layout.tsx` imports `@aws-amplify/ui-react/styles.css` AFTER `globals.css`, the default theme overrides our `@theme` block's color tokens.
**Why it happens:** the Amplify default stylesheet uses `:root` selectors, which ALSO affect our `--color-hey-bg` etc. — except they don't (different namespace). The risk is bidirectional: our tokens leak into Amplify (no — different namespaces) AND Amplify defaults survive into our theme (no — `<ThemeProvider>` `data-amplify-theme="hey-banco"` selector wins).
**How to avoid:**
1. Keep the import order from Phase 1: `app/layout.tsx` imports `./globals.css` THEN `@aws-amplify/ui-react/styles.css` (already correct).
2. Verify in DevTools: open `/login`, inspect a `<Authenticator>` input, check that computed `background-color` resolves to `#000000` (our token), not `white` (Amplify default).
**Warning signs:** Authenticator inputs render with white background despite the theme — means the `<style data-amplify-theme>` block didn't mount. Check that `<ThemeProvider>` is wrapping `<Authenticator>` (not above it in a different render branch).

### Pitfall 2: Importing `motion` from the wrong path

**What goes wrong:** `import { motion } from "framer-motion"` works (since `framer-motion@12.38` is published as a back-compat alias), but mixing both imports in different files leads to two copies of the motion runtime in the bundle, ~120kb double-up.
**Why it happens:** training data still says `"framer-motion"`. Code reviewers may not catch.
**How to avoid:** `bun add motion@^12.38.0` (NOT `framer-motion`). Add an ESLint `no-restricted-imports` rule (optional):
```js
// eslint.config.* — restrict the old name
{
  rules: {
    "no-restricted-imports": ["error", { paths: [{ name: "framer-motion", message: "Use motion/react instead" }] }],
  },
}
```
**Warning signs:** Bundle analyzer (Phase 4 territory) shows two `framer-motion` chunks. For Phase 03.1: simple grep `grep -r "framer-motion" app/` should return zero matches once installed.

### Pitfall 3: `useSearchParams` Suspense boundary regression in `/login`

**What goes wrong:** Phase 3 `app/login/page.tsx` already handles this via outer `<Suspense>` + inner `LoginPageInner`. Phase 03.1's modification (wrap `<Authenticator>` in `<ThemeProvider>`) MUST keep the Suspense split. If a developer collapses the file back to a single component during refactor, `bun run build` fails with "Missing Suspense boundary with useSearchParams" (Phase 3 K-3 / L-1).
**Why it happens:** the wrap looks like a small change, "let me also clean up the Suspense split" instinct.
**How to avoid:** Plan-checker greps for `useSearchParams` in `app/login/page.tsx` — must have a `Suspense` import and JSX usage in the same file (or in an importer).
**Warning signs:** dev passes, `bun run build` fails. Catch in the per-plan static gauntlet, not at phase close.

### Pitfall 4: Reduced motion preference ignored

**What goes wrong:** D-51 enumerates several animations. Users with `prefers-reduced-motion: reduce` see vestibular-triggering motion if we don't respect the setting.
**Why it happens:** easy to forget; the chat UI animations are subtle and the developer doesn't think of accessibility.
**How to avoid:** Wrap motion components in `useReducedMotion()` checks (Pattern 3) OR use the global `<MotionConfig reducedMotion="user">` once at the top of `/app` (motion@12 supports this — it disables transform/layout animations system-wide, preserving opacity). HAVI ring CSS already has `@media (prefers-reduced-motion: reduce)` (Pattern 4).
**Warning signs:** reviewer checks `prefers-reduced-motion` in DevTools (Rendering panel → "Emulate CSS media feature: prefers-reduced-motion: reduce"). All chat motion should still feel natural (fades preserved, slides removed).

### Pitfall 5: `next/font` weight array vs `'variable'` mismatch — IBM Plex Serif is non-variable

**What goes wrong:** Code that passes `weight: 'variable'` to `IBM_Plex_Serif` will fail TypeScript: the type signature for Serif (verified in `node_modules/next/dist/compiled/@next/font/dist/google/index.d.ts`) is `weight: '100' | '200' | '300' | '400' | '500' | '600' | '700' | Array<…>` — NO `'variable'` option. **IBM Plex Sans HAS `'variable'`; IBM Plex Serif does NOT.**
**Why it happens:** copy-paste error from Sans config.
**How to avoid:** explicit weight array `weight: ["400", "700"]` for Serif (D-45). For Sans, either explicit array `["400", "500", "700"]` or `weight: 'variable'` (which subsets the variable axis but loads the full file — slightly larger, more flexible). D-45 says explicit array; honor it.
**Warning signs:** `bun run typecheck` errors on `Type '"variable"' is not assignable to type '"100" | "200" | …'` — exact message tells you which font.

### Pitfall 6: `app/app/layout.tsx` accidentally modified

**What goes wrong:** the file is `"use client"` + `<AuthGuard>{children}</AuthGuard>`. A developer adding a wordmark / app shell at top of `/app/*` may put it in the layout, "for reuse." That breaks Phase 3 D-30's contract that the layout is reusable across any future `/app/*` sub-route. More importantly, putting more JSX in the layout means motion/AnimatePresence in `template.tsx` won't see the right boundaries.
**Why it happens:** layout vs template confusion. "Layout = the visual frame" is a common mental model from non-App-Router frameworks.
**How to avoid:** **DO NOT MODIFY `app/app/layout.tsx`.** App shell (header with HAVI ring, back button) is a per-page component (`<AppHeader>`). If a true persistent shell is wanted, use `app/app/template.tsx` (re-mounts on segment change but preserves outer layout) — but D-46 says each page has its own header, so per-page is correct.
**Warning signs:** plan touching `app/app/layout.tsx` should fail plan-check.

### Pitfall 7: Mock data type drift

**What goes wrong:** `app/_data/mock.ts` exports `mockTransactions: Transaction[]`. Components import the array and access `t.merchant`, `t.amount`, etc. As components are scaffolded, the field names drift (one expects `name`, another `merchant`). When mock is later swapped for real fetch, the surface is wide.
**Why it happens:** parallel component implementation without locked types.
**How to avoid:** define types in `app/_data/types.ts` FIRST (Wave 0 task), import from there in BOTH `mock.ts` and components. Lock fields: `Transaction = { id, date, merchant, amount, categoryId, iconHint?, color? }`.
**Warning signs:** TS errors in component when changing `mock.ts`; or component renders empty cells. TypeScript catches both at typecheck.

### Pitfall 8: `Hub.listen` returns a cleanup fn, not a Promise

**What goes wrong:** `useEffect(() => { Hub.listen(…); return () => { /* nothing */ } }, [])` leaks the listener. Multiple mounts (StrictMode dev) compound.
**Why it happens:** Hub API is unfamiliar; developer may not check return type.
**How to avoid:** **always** `return Hub.listen(channel, cb)` directly from the effect — Hub.listen returns `StopListenerCallback` which is exactly the `() => void` shape useEffect cleanup wants. Pattern 6 shows this.
**Warning signs:** in dev, sign in → sign out → sign in → sign out — observe console: more listeners attached each cycle (logged via custom debug).

### Pitfall 9: `prefers-color-scheme: dark` block in current `globals.css`

**What goes wrong:** the existing `globals.css` has `@media (prefers-color-scheme: dark) { :root { --background: #0a0a0a; --foreground: #ededed; } }`. After Phase 03.1 we are dark-only — but a user on light-mode OS will see THE OLD `--background: #ffffff` from the `:root` block above. We're replacing all this with the hey palette.
**Why it happens:** failure to fully replace `:root` from Phase 1 scaffold.
**How to avoid:** `app/globals.css` Phase 03.1 rewrite: REMOVE the entire `:root { --background: #ffffff; ... }` block AND the `@media (prefers-color-scheme: dark) { ... }` block. REMOVE the `body { background: var(--background); color: var(--foreground); font-family: Arial, ... }` rule (Tailwind utility classes on `<body>` in `layout.tsx` replace it). Replace with `@theme inline {…}` + `@theme {…}` + minimal body reset.
**Warning signs:** light-mode OS user opens `/`, sees flash of white before dark palette kicks in. Or `/login` Authenticator card appears with default white surface despite `<ThemeProvider>`.

### Pitfall 10: Patrón regex catch-all order

**What goes wrong:** placing `/.*/ ` (catch-all) BEFORE specific patterns means HAVI never matches anything specific. Always falls back.
**Why it happens:** ordering arbitrarily during pattern evolution.
**How to avoid:** ESLint custom rule (overkill) OR a comment-banner in `patterns.ts`: `// CRITICAL: catch-all /.*/ MUST be the last entry. Specific patterns above.` Plan-checker can grep `app/_data/patterns.ts` for `match: /\.\*/` and verify it's the last array entry.
**Warning signs:** every chat input triggers fallback. Manual flow walkthrough catches it.

```tsx
"use client";
import { useEffect, useRef, useState } from "react";
import { useAuthenticator } from "@aws-amplify/ui-react";
import { fetchUserAttributes } from "aws-amplify/auth";
import type { ChatMessage } from "@/app/_data/types";
import { haviPatterns } from "@/app/_data/patterns";
import { mockBalance, mockSpentThisMonth, mockSpentPct, mockSparkline } from "@/app/_data/mock";
import { loadChat, saveChat, useSignOutClear } from "@/app/_hooks/useChatPersistence";
// component imports (Composer, MessagesList, AppHeader, etc.) go here

const FIRST_NAME_KEY = "havi-first-name-v1";

function deriveFirstName(name?: string, email?: string): string {
  if (name) return name.split(" ")[0];
  if (email) return email.split("@")[0];
  return "amigo";
}

export default function ChatHome() {
  const { user } = useAuthenticator(({ user }) => [user]);
  const email = user?.signInDetails?.loginId;
  const [firstName, setFirstName] = useState<string>(() => {
    if (typeof window === "undefined") return "amigo";
    return localStorage.getItem(FIRST_NAME_KEY) ?? "amigo";
  });

  // Refresh firstName from Cognito (background)
  useEffect(() => {
    let cancelled = false;
    (async () => {
      try {
        const attrs = await fetchUserAttributes();
        const fn = deriveFirstName(attrs.name, attrs.email ?? email);
        if (!cancelled) {
          setFirstName(fn);
          localStorage.setItem(FIRST_NAME_KEY, fn);
        }
      } catch (e) {
        if (process.env.NODE_ENV !== "production") {
          console.warn("[ChatHome] fetchUserAttributes failed:", e);
        }
      }
    })();
    return () => { cancelled = true; };
  }, [email]);

  // Hydrate messages from localStorage or seed with intro+snapshot
  const [messages, setMessages] = useState<ChatMessage[]>(() => {
    const persisted = typeof window !== "undefined" ? loadChat() : null;
    if (persisted && persisted.length > 0) return persisted;
    return seedIntroMessages(firstName);
  });
  const [typing, setTyping] = useState(false);

  // Re-seed greeting if firstName changed AND chat is at intro state only
  useEffect(() => {
    setMessages((prev) => {
      if (prev.length > 0 && prev[0].kind === "intro") {
        const updated = [...prev];
        updated[0] = { ...prev[0], text: `hey, ${firstName}.` };
        return updated;
      }
      return prev;
    });
  }, [firstName]);

  // Persist (debounced)
  const debounceRef = useRef<ReturnType<typeof setTimeout> | null>(null);
  useEffect(() => {
    if (debounceRef.current) clearTimeout(debounceRef.current);
    debounceRef.current = setTimeout(() => saveChat(messages), 250);
    return () => { if (debounceRef.current) clearTimeout(debounceRef.current); };
  }, [messages]);

  // Sign-out clears localStorage (Hub.listen)
  useSignOutClear();

  function send(text: string) {
    const trimmed = text.trim();
    if (!trimmed) return;
    const userMsg: ChatMessage = {
      id: crypto.randomUUID(),
      from: "user",
      kind: "user-text",
      text: trimmed,
      createdAt: new Date().toISOString(),
    };
    setMessages((m) => [...m, userMsg]);
    setTyping(true);

    const matched = haviPatterns.find((p) =>
      typeof p.match === "function" ? p.match(trimmed) : p.match.test(trimmed)
    )!; // catch-all guarantees match
    const response = matched.respond(trimmed);
    const delay = response.delayMs ?? 800 + Math.random() * 700;

    setTimeout(() => {
      setTyping(false);
      setMessages((m) => [...m, ...response.messages]);
    }, delay);
  }

  return (
    <main className="hey-app-frame flex min-h-screen flex-col">
      {/* AppHeader (HAVI ring + label) */}
      {/* MessagesList (renders messages by kind, includes TypingIndicator when typing) */}
      {/* SuggestionChips (4 quick replies) */}
      {/* Composer (input + send button) */}
    </main>
  );
}

function seedIntroMessages(firstName: string): ChatMessage[] {
  const now = new Date().toISOString();
  return [
    { id: "intro-1", from: "havi", kind: "intro", text: `hey, ${firstName}.`, createdAt: now },
    {
      id: "snapshot-1",
      from: "havi",
      kind: "snapshot",
      data: {
        balance: mockBalance,
        spent: mockSpentThisMonth,
        spentPct: mockSpentPct,
        topCategory: "Comida",
        sparkline: mockSparkline,
      },
      createdAt: now,
    },
  ];
}
```

| `next/font` weight-array required for Geist | `next/font` `Geist` is variable; `IBM_Plex_Sans` supports `'variable'`; `IBM_Plex_Serif` is non-variable (must list weights) | n/a (Plex never had a TrueType-variable serif) | Phase 03.1: use `weight: ["400", "700"]` for Serif explicitly |
| `useReducedMotion` from `framer-motion` | `useReducedMotion` from `motion/react` | mid-2025 | identical API |
| `Amplify.configure(outputs, { ssr: true })` (Next.js docs recommendation) | `Amplify.configure(outputs)` (no options) — until `@aws-amplify/adapter-nextjs` is installed | Phase 2 walk-back (RESEARCH L-3) | Phase 03.1: do NOT regress the walk-back |
| `framer-motion` `exitBeforeEnter` | `<AnimatePresence mode="wait">` | Framer Motion v10 | Phase 03.1: use `mode="wait"` |
| Hand-rolled CSS rotation for HAVI ring | CSS `conic-gradient` with `@keyframes rotate` | Conic-gradient browser baseline (Safari 16.4 = March 2023) | Phase 03.1: CSS-only HAVI ring, zero JS |

**Deprecated / outdated (do NOT use):**
- `import { motion } from "framer-motion"` — works (alias), but new code should use `motion/react`
- `tailwind.config.js` `theme.extend` — Tailwind 4 doesn't have a JS config in this project (`@tailwindcss/postcss` only)
- `<Authenticator theme={...}>` — does not exist in v6 type (verified `Authenticator.d.ts:8-18`)
- `localStorage` access without `typeof window !== "undefined"` check — SSR unsafe
